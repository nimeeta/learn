Ericsson Case Study

How might we create a design solution for monitoring deployed services in Kubernetes?


About Ericsson

Ericsson is one of the leading providers of Information and Communication Technology (ICT) to service providers.They enable the full value of connectivity by creating game-changing technology and services that are easy to use, adopt, and scale, making their customers successful in a fully connected world.

Problem Statement

The Business use case associated with this task was to create a platform for the user to monitor the performance of services deployed on Kubernetes in the best optimal way. Performance monitoring is crucial to make sure that all applications are always up and running.

Solution

Interactive prototype to monitor services and pods under each service, dependencies between services deployed on Kubernetes.

Responsibilites

  • As part of my internship, I was responsible for designing a functional service monitoring UI by understanding the user needs and cloud-native applications design methodologies.
  • I collaborated with other designers to prepare the UX design document aligned with the business use-cases requirements.
  • Conducted workshops and feedback sessions internally to understand the business flow and overall user experience.
  • Installed Kubernetes cluster locally on the machine to visualize and design the prototype with real experience.

Time Frame

4 months

Tools Used
  • Adobe XD
  • Adobe Photoshop
  • Ubuntu 18.04 app
  • Docker Desktop app
  • Kubernetes

UX Process

During this project, I used the design thinking process to get the task at hand done. I received a document mentioning all the business requirements. It was expected of me to design the GUI by analyzing the user requirements with this BUC.

Persona

So to analyze the user requirement, I put myself in the user's shoes gone through the process of deploying services and installing applications on my local machine. I tried monitoring service level KPIs.It helped me to empathize with the user by understanding their needs.

Based on my insights, I created a UX analysis document to acknowledge user requirements with the BUC(Business use case).

Ideation-Solutions Brainstorm

In order to start ideation, I researched more about Kubernetes . I learned about the components of Kubernetes and their functionality, relationships with different components, and how these are interconnected.

Purpose of Service UI

The main purpose of this GUI was to provide a service-level monitoring interface that can be accessed from the central Operations and Maintenance dashboard. There is also a possibility to access this GUI independently from the service app.

The service monitoring UI will provide the visualization of data required to monitor services hosted on the Kubernetes platform. This will help the system engineer to monitor the performance of each service.

In addition to that, the same GUI can be used to monitor the dependencies between the services running in the same or different clusters.

Iterations

In my first iteration, I focused on the priority of information for the user. I decided to show alarms at the service level so that the user will quickly be able to drill further down to analyze the problem accurately. By drilling a step further in the UI, the user will see all pod statuses. Then further drilling down into POD level, suggested to show KPIs like CPU usage, outbound, inbound Network traffic, and memory usage).

The user can also view the dependencies between services using switches to change view from card to relation graph view. At the relation graph view, the user will still be able to view alarms as high-priority information and can drill down in UI.

Final Solution

For the final solution, I had series of discussions and workshops with the developer of the team and with the core UX team. I received feedback that there will be more KPIs at the service level that needs to accommodate in the current design.

So in the second iteration, the user can view the high-priority information in card or graph view. By drilling down a step in UI the user can view a detailed alarm graph with other KPIs(Logs and distributed traces) in the widgets.By expanding the widgets user can access other UI for further analyzing the problem.


Takeaways

During the internship, I learned about cloud technology and micro-service-based architecture and met new people from different fields of profession. I learned about the role and responsibilities of UX designer in Ericsson. I got good exposure to working cultures between teams within the UX field and developers.
